# Changelog

Concise log of non-trivial vault system changes, newest first. One line per entry: date, time, what changed and why.

- 2026-05-09 — **Hyperspace migration: backbone lock-in.** Trimmed the migration scaffold to a true backbone after design discussion with the user. (1) Reverted Workspace top-level files to human style (Title Case, spaces allowed) — `Home.md`, `Inbox.md`, `Task Capture.md`, `Map.md`. `_agentic` keeps tech style (kebab-case + typed suffixes). (2) Dropped the `.MANAGED` suffix on managed files; the `Managed/` folder already encodes the operator-ownership boundary, so `Tasks.MANAGED.md → Tasks.md`, `Calendar.MANAGED.md → Calendar.md`. (3) Collapsed `_agentic/core-rules/` (seven one-line placeholder files) into a single `_agentic/conventions.CORE.md` with sections for routing, permissions, file naming, archive, logs. The `core-rules/` folder is gone; future splits (`router.CORE.md`, `permissions.CORE.md`, etc.) are pre-declared as deferred modules with explicit graduation triggers, but only added when concrete pressure forces them. (4) Made `AGENTS.md` the single registry for active operators — adding/removing an operator now means dropping a folder + adding one row to its table; no other file is edited. Eliminates the prior ripple-edit problem across `AGENTS.md` + multiple `core-rules/` files. (5) Dropped the planned `_agentic/tools/` folder and `registry.TOOL.jsonc` — zero tools currently exist; deferred until two operators define the same tool, or a tool needs permission rules, or capability discovery becomes painful. (6) Removed `migration/drafting rules.md` (its one rule about operator activation is now in `conventions.CORE.md §Operator activation`). Also removed two empty placeholder files (`_agentic/operators/overview.OPERATOR.md`, `_agentic/archive/overview.ARCHIVE.md`) consistent with the newly explicit growth rule: "The empty-placeholder pattern is forbidden." (7) Rewrote `Hyperspace Agreement.md` end-to-end to match the new shape, including a Deferred Modules table (with graduation triggers) and an explicit Growth Rule. Net effect: the backbone is the smallest set of files that gives unambiguous answers to "where does this go?" and "who owns it?" Adding an operator is now a one-edit operation. Open items: drain contract for `Task Capture.md` (blocks task-manager implementation, not the backbone) and the human-readable schema for `Managed/Tasks.md` and `Managed/Calendar.md`.
- 2026-05-09 — **Hyperspace migration: capture-queue split + kebab-case Workspace.** Applied agreed structural changes to `migration/Workspace/` after design review with the user. (1) Resolved the Tasks.md / Tasks.MANAGED.md duality by adopting the drain pattern: `Tasks.md` renamed to `tasks-capture.md` — a write-only capture queue that the task-manager operator drains into `Managed/Tasks.MANAGED.md` (which becomes the human-readable read surface for the active task list). (2) Deleted `Calendar.md` — calendar items normally arrive via import, so a freeform calendar capture surface would be an imaginary department; rare freeform calendar mentions go through `inbox.md` and the signal-router. (3) Renamed Workspace top-level files to kebab-case for consistency with `_agentic` and to drop spaces from filenames: `Inbox.md → inbox.md`, `Home.md → home.md`, `Map.md → map.md`, `Workspace Overview.md → workspace-overview.md`. The `Managed/` folder and `*.MANAGED.md` suffixes are unchanged. (4) Updated `Hyperspace Agreement.md` to reflect the new backbone, the capture-queue role, the kebab-case rule, and the "Where things go" table; added explicit notes that managed files must be human-readable and that the drain contract for `tasks-capture.md` is the next concrete spec to write before the task-manager operator can be implemented. Rationale: the original Tasks.md / Tasks.MANAGED.md duality was a deferred decision that would have caused desync the first time a task was added in a hurry; the drain pattern gives clean ownership semantics (User writes only to capture, operator writes only to managed, drain is the one touchpoint).

- 2026-04-23 — **Vault-check exemption refactor (config v2.4).** Introduced a shared `rules.exempt_dirs` config key (default `["_archive", "_secretary"]`), used by **L3** (slug uniqueness), **O1** (outline reachability), and **M6** (recommended frontmatter). `outline.exempt_dirs` is kept as a fallback for backward compat but is no longer needed. `vault_check.py` changes: (a) added `get_exempt_dirs(cfg)` helper reading `rules.exempt_dirs` with fallback to `outline.exempt_dirs`; (b) L3 now only counts **non-exempt** paths as collisions — a non-exempt file is not flagged when its only "duplicate" slugs live in exempt dirs, and collisions entirely within exempt dirs never fire; (c) `check_frontmatter` gained an `emit_m6` flag; M6 is suppressed for files under `rules.exempt_dirs`, for files listed in `append_only.files` (inbox.md, todo.md), and for truly empty files (0 non-whitespace bytes); M1/M2/M4 still fire as before (structural errors). - 2026-04-23 — **New feature: `fix-frontmatter`.** Added `_meta/features/fix-frontmatter/` (stdlib-only Python, idempotent, dry-run by default). Backfills missing `title` and `created` keys on notes flagged by M6. Inference chain: `title` = first H1 → filename stem; `created` = `git log --diff-filter=A --follow --format=%as` → OS `st_birthtime` → today. Preserves existing frontmatter verbatim (only adds missing keys). Skips protected dirs, `_archive/`, `_secretary/`, and empty files. Registered in `_meta/features/registry.jsonc` with triggers `fix frontmatter / auto-correct frontmatter / backfill frontmatter / add missing frontmatter`, available to `admin` and `maintenance` modes. Test suite `test_fix_frontmatter.py` — 27 tests covering parse, infer, build, process, walk, and git-add-date fallback. All tests pass. Initial vault run: scanned 652 files, updated 628, 23 already complete, 1 empty-file skip. Vault-check after: warnings 1364 → 112 (remaining 56 files × 2 keys are all inside `_archive/`/`_secretary/`, which the script intentionally skips); errors unchanged at 10 (all pre-existing). No git operations performed by the feature — user reviews `git diff` and commits manually. **Amendment:** the initial run prepended frontmatter to `inbox.md` and `todo.md`, violating the append-only contract from §1b. Reverted those two files and patched `walk_vault()` to read `append_only.files` from config and skip them. Added regression test `test_skips_append_only_files`. Full test suite 28/28 passes. Re-run is now idempotent and leaves inbox/todo untouched.
