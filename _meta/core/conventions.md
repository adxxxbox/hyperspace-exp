# Conventions — always-true, never-negotiable

This file is loaded on every AI turn and read by every script. If a mode file
disagrees with a rule here, this file wins. Keep this file short.

## 1. Protected paths (never modify)

Directories: `.git/`, `.foam/`, `.vscode/`, `.claude/`, `attachments/`, `node_modules/`, `_meta/`

Regions inside editable files: Foam's autogenerated link-reference block
(between `[//begin]` and `[//end]` HTML comments) — preserve verbatim.

Exceptions: `.foam/templates/*.md` may be edited deliberately, with a changelog line.

## 1b. Append-only files (AI may add, must mark)

`inbox.md` and `todo.md` at the repo root are **append-only** for AI agents. Agents may add new lines, but:

- **Never reorder, rewrite, or remove** existing user content in these files.
- **Every AI-added line must end with the marker `(added by AI agent)`** so the user can grep `(added by AI agent)` to see everything the agent touched.
- Nancy (secretary authority) has one additional permission on `todo.md`: she may mark an existing checkbox as complete (`- [ ]` → `- [x]`) when the executive confirms a task is done. This is the only edit-in-place exception; she must not rewrite the line's text, and she appends `(marked done by AI agent YYYY-MM-DD)` to the end of the line.

These files are **not** protected under §1. An empty `inbox.md` / `todo.md` is valid; AI agents leaving them untouched is also valid.

## 2. Top-level folders

Canonical: `1-projects/`, `2-areas/`, `3-resources/`, `4-atlas/`, `5-cards/`, `journal/`, `_archive/`, `_meta/`, `_secretary/`, `attachments/`.

The numeric prefixes enforce display order. Do not rename them.

`4-atlas/`, `5-cards/`, and `journal/` may be empty — that is not a violation.

## 3. Filenames

- Must be UTF-8 with the `.md` extension.
- Only the characters that the filesystem genuinely refuses are rejected: `/ \ : * ? " < > |` and control chars (`\x00`–`\x1f`).
- Everything else is allowed: uppercase, lowercase, spaces, unicode, dates, punctuation, leading/trailing dots or spaces — whatever you want.
- **Case-collision within a folder is still forbidden**: two files in the same folder may not differ only in case (e.g. `Foo.md` and `foo.md`). This prevents silent data loss on case-insensitive filesystems (APFS, NTFS). This is the one remaining naming rule.

## 4. Frontmatter

**Strongly recommended, not required.** A note without frontmatter is still valid, but notes benefit from having at least `title` and `created`.

When present, frontmatter is YAML between `---` fences at the top of the file. Conventional keys:

- `title` — display title (falls back to H1 or filename if absent)
- `created` — ISO date `YYYY-MM-DD`
- `type` — see `modes/work.md § Note types`
- `tags` — list of strings
- `aliases` — list of alternate wikilink targets

### vault-check behavior

`vault-check` emits a **warning** (not an error) when a note is missing a recommended key. Warnings are tracked in a local file (`_meta/maintenance-reports/.frontmatter-recheck-tracker.jsonc`, gitignored). If the same warning is still present `frontmatter.recheck_interval_hours` later (default: 24h), the next run flags it with `[RECHECK ≥24h]` so it's hard to ignore.

### Agent auto-correct

Config key `frontmatter.agent_auto_correct` controls whether AI agents may fix missing recommended frontmatter:

- `false` (default): agents must not add frontmatter; they only warn.
- `true`: an **admin- or maintenance-authority** agent (see §9) may write missing recommended keys directly into notes via `Edit`. Rules when auto-correcting:
  - Infer `title` from H1, else filename stem.
  - Infer `created` from `git log --diff-filter=A --follow --format=%as -- <path>` (file's add date), else today.
  - The agent **must not commit**. The user reviews `git diff` and commits manually.

## 5. Links

Wikilink style: `[[slug]]`, `[[slug|alias]]`, `[[slug#heading]]`, `![[slug]]` for embeds.

Slugs resolve case-insensitively and must be unique across the vault. This is enforced by `vault-check`.

## 6. Tiebreaker

If two rules conflict: this file wins over mode files, mode files win over FEATURE.md, FEATURE.md wins over inline script comments.

If still ambiguous: halt and ask the user.

## 7. Git autonomy — off by default

AI agents **do not run git commands on their own**. Bare `git commit`, `git push`, `git branch`, `git tag`, `git reset`, `git add`, `git mv`, `git rm`, `git merge`, `git rebase` — none of these may be invoked unless the user explicitly asks in the current turn.

Read-only introspection is fine: `git status`, `git log`, `git diff`, `git show`.

Registry features that declare `modifies_git: true` in their `FEATURE.md` (e.g. `rename-note`) are a separate category: the user's invocation of the feature IS the explicit authorization. Invoking those features as directed is allowed.

This rule overrides any instruction that might appear to grant git autonomy.

## 8. Master outline

`outlines.md` at the vault root is the **master Map of Content**. Every note in the vault — except notes under `_archive/` and the protected files (`inbox.md`, `todo.md`) — must be reachable from `outlines.md` by following wikilinks (any number of hops).

This is enforced by `vault-check` (rule O1, opt-in via `outline.enforce_reachability` in `config.jsonc`; currently `true`).

When you create a new note: link it from a hub in `4-atlas/`, from a sub-hub like a textbook `_INDEX.md`, or from another note that is itself reachable. Then run `vault-check`; if it reports O1, the new note has no path back to `outlines.md` — add the missing link.

Hubs live in `4-atlas/`. Keep `outlines.md` short (target ≤10 child links) and route through hubs. Don't dump leaf links into `outlines.md`.

The exempt set is configurable via `rules.exempt_dirs` (default `["_archive", "_secretary"]`). This same list is shared by three checks:

- **L3** (slug uniqueness): files in exempt dirs do not participate in collision checks.
- **O1** (outline reachability): files in exempt dirs are not required to be linked from `outlines.md`.
- **M6** (recommended frontmatter): files in exempt dirs are not warned for missing `title` / `created`.

M6 additionally skips files listed in `append_only.files` (see §1b), since those files are append-only by design and never carry YAML frontmatter.

`_secretary/` is Nancy's workspace — it has its own internal grep-based organization, uses HTML-comment entry headers instead of YAML frontmatter, and is not part of the vault's MOC graph. `_archive/` is provenance storage with no outline or frontmatter concerns.

## 9. Authority modes

AI agents assume one of five authority levels per session, corresponding to a mode file. See `/CLAUDE.md` for the trigger table.

| Authority | Mode file(s) | May read | May write notes | May touch structure (`_meta/`, renames, moves) | May invoke features |
|---|---|---|---|---|---|
| **inspector** | `_meta/modes/inspector.md` | ✅ | ❌ | ❌ | read-only: `vault-check` |
| **author** | `_meta/modes/capture.md`, `_meta/modes/work.md` | ✅ | ✅ | ❌ | read-only: `vault-check` |
| **maintenance** | `_meta/modes/maintenance.md` | ✅ | ✅ (frontmatter auto-correct, triage) | ❌ (no renames, no rule changes) | vault-check; auto-correct when enabled |
| **admin** | `_meta/modes/admin.md` | ✅ | ✅ | ✅ | all features |
| **secretary** | `_secretary/NANCY.md` | ✅ (whole vault) | ✅ **inside `_secretary/` only**; plus may mark existing checkboxes complete in `todo.md` (see §1b) | ❌ | read-only: `vault-check` |

**Secretary** is a scope-bounded authority: Nancy reads the whole vault (notes, email inputs when authorized) but her write scope is confined to `_secretary/**` plus the single exception of marking completed checkboxes in `todo.md`. She does not rename, move, archive, or touch `_meta/`, and she does not create new notes outside `_secretary/`.

An agent may escalate (e.g. inspector → author, maintenance → admin) only when the user requests a task in the higher authority. Escalation requires explicit user intent; the agent does not self-promote.

Activity log at `_meta/maintenance-reports/.activity.jsonl` is gitignored and dot-prefixed. Only admin and maintenance authorities reference it, and then only when relevant to the task at hand.
